# FVM 模拟器：基于 WebSockets 的像素流引擎全栈架构与部署指南

本指南详细说明了将基于 C++/OpenGL/CUDA 的本地模拟器改造为 C/S 像素流（Pixel Streaming）架构的技术原理。适合有一定前后端开发经验的开发者（如熟悉 Java、Vue3 等环境）参考和学习。

## 一、 架构路线与技术选型

### 1.1 数据流架构 vs 像素流架构
常规的分布式应用设计会将计算（Solver）和表现（Renderer）分离。在这种模式下，服务端负责模拟流场计算，并通过网络将含有物理量的矩阵数据下发到客户端，客户端使用 WebGL 进行呈现。
*   **带宽瓶颈**：高精度网格（如 1000x1000）包含大量浮点数，若以 60 FPS 传输全量场数据，网络吞吐量将达到数百 MB/s，远超常规千兆局域网的处理上限。
*   **像素流架构（Pixel Streaming）**：计算与渲染统一在配置独立显卡的 Windows 服务端进行。借助 OpenGL API 获取栅格化后的帧缓存数据（Framebuffer），通过极低的 CPU 开销将其压缩为图像编码格式（JPEG），通过 WebSocket 长连接协议推流给客户端。客户端仅作为一个全屏的 HTML5 `<canvas>` 用作显示流容器，同时作为外部交互输入事件的采集端向服务端回传操作坐标与指令。

### 1.2 核心技术栈
*   **通信与进程模型**：引入 `Mongoose`。这是一个单文件的事件驱动跨平台网络库，能够同时承载 HTTP 端点响应和管理非阻塞的 WebSocket 并发长连接。
*   **数据序列化**：引入 `nlohmann/json`，解析前端发送过来的交互数据对象。
*   **图像编码**：引入 `stb_image_write`，直接读取内存地址中的 RGBA 原始点阵数据，利用哈夫曼编码与离散余弦变换算法序列化为 JPEG 二进制格式，全程无磁盘 I/O。

## 二、 核心代码模块详细解析

### 2.1 极简前端与 Web 端渲染实现 (`web_server.cpp`)
本架构摒弃了传统的前后端拆分，将提供给客户端的静态资源直接作为字符串常量，在服务端截获根路由 `/` 的请求时下行返回。
*   **双工通信升级**：JavaScript 中建立 `WebSocket` 实例，核心是将参数 `binaryType` 设置为了 `'blob'`。这告知浏览器引擎网络流传来的是经过打包的二进制对象，而非通常的 UTF-8 明文数据。
*   **画面重绘流水线**：前端完全不需要引入 WebGL 环境，所有工作交由原生的 Canvas 2D 上下文完成：
    ```javascript
    ws.onmessage = async (e) => {
        if (e.data instanceof Blob) {
            // 通过内置 API 生成指向二进制堆的内部映射 URL
            const img = new Image();
            img.onload = () => {
                // 侦听图像维度，若服务端分辨率或缩放产生变化，重置画布标尺
                if (canvas.width !== img.width || canvas.height !== img.height) {
                    canvas.width = img.width; canvas.height = img.height;
                }
                // 在起始坐标点 (0,0) 一次性将解码结束后的整帧栅格数据绘制上去
                ctx.drawImage(img, 0, 0); 
                // 务必释放引用句柄，促使垃圾回收（Garbage Collection）执行，防止由于内存溢出导致浏览器崩溃
                URL.revokeObjectURL(img.src); 
            };
            img.src = URL.createObjectURL(e.data); // 为对象赋值 URL 触发异步解析回调
        }
    };
    ```
*   **交互事件归一化处理**：通过坐标空间映射保证跨物理设备的分辨率无关性：
    ```javascript
    // e.clientX 包含绝对屏幕坐标，减去画布相对边距后，除以画布总宽度或高度。
    // 计算出的 px/py 始终会落在区间 [0.0, 1.0] 的相对浮点型数据内。
    const px = (e.clientX - rect.left) / rect.width;
    const py = (e.clientY - rect.top) / rect.height;
    ```

### 2.2 后端帧缓冲捕获与图像压缩 (`web_server.cpp` -> `WebServer_CaptureFrame`)
此函数负责在显卡输出的渲染管线末端介入拦截数据。
*   **Framebuffer 点阵读取**：调用 OpenGL 的函数 `glReadPixels(0, 0, width, height, GL_RGBA, GL_UNSIGNED_BYTE, pixels.data());`。该调用挂起图形队列直到像素总线写入到指定的主存地址。
*   **坐标系矩阵反转**：需注意，OpenGL 使用以**左下角**为坐标原点 `(0,0)` 的笛卡尔空间体系；而 JPEG 编码层遵循计算机图像领域的**左上角**原点体系。为避免图像上下颠倒，必须在按行地址进行内存倒序（垂直翻转）：
    ```cpp
    int row_bytes = width * 4; // 1 个 RGBA 像素等于 4 字节
    for (int y = 0; y < height; ++y) {
        memcpy(&flipped_pixels[y * row_bytes], 
               &pixels[(height - 1 - y) * row_bytes], row_bytes);
    }
    ```
*   **无落盘编码派发**：利用 `stbi_write_jpg_to_func` 挂载函数指针回调。其生成的编码字节会持续地被追加写入一个 `std::vector<uint8_t>` 对象返回，并交由推流调度。

### 2.3 渲染循环介入与状态反向注入 (`main.cpp`)
改造的核心是将独立进程化的网络轮询模型嵌入到原有的 `glfw`（主线程图像系统）的生命周期循环中，从而避免在图形引擎与网络引擎中维护复杂且极易引发竞态的局部锁：
*   **事件拉取策略**：在单层循环内部使用 `WebServer_PollEvents()`，指派 Mongoose 利用 CPU 闲散周期执行底层的 Socket 的状态探测（非阻塞读取）。
*   **绝对事件位置还原与数据结构注入**：取出经由前文转换传来的相对映射坐标，在后端还原成逻辑绝对像素系，进行 UI 的篡改触发和拦截。
    ```cpp
    // 取得真正的宿主执行时所建立的原生图形窗口的长宽属性
    int w, h;
    glfwGetWindowSize(window, &w, &h);
    // 反解算获取实际逻辑像素
    int map_x = static_cast<int>(px * w);
    int map_y = static_cast<int>(py * h);
    
    // 加载 ImGui 组件库的状态接口，将刚刚计算完的鼠标特征属性无痕替换写入全局 IO 接口中。
    ImGuiIO& cur_io = ImGui::GetIO();
    cur_io.AddMousePosEvent(static_cast<float>(map_x), static_cast<float>(map_y));
    // 监听按键释放态标志
    if (type == "mousedown") cur_io.AddMouseButtonEvent(button, true);
    ```
*   **流式下行发送与降采样限流**：
    由于每一帧图形操作后直接截获极度消耗算力且传输带宽极易膨胀。若项目主体渲染运行在 `60FPS` 以上的高刷新率体系内，进行减位采样（降频）。
    ```cpp
    static int frameSkip = 0;
    // 使用 WebServer_HasClients 标记避免空载运算。
    // frameSkip 记录运行频率限制为 1/2 帧率推送，保证终端平滑度的情况下减少负荷。
    if (WebServer_HasClients() && (++frameSkip % 2 == 0)) {
        // ...执行CaptureFrame 后调用 WebServer_BroadcastFrame 以 WEBSOCKET_OP_BINARY 的指令规范发送出流。
    }
    ```

## 三、 本地局域网部署与测试执行教程

你需要使得包含模拟器的主计算节点（Windows PC）和终端浏览器所在的设备（MacBook）建立双端 TCP 通讯。请严格按照以下步骤配置连接：

### 3.1 获取运行侧（Windows 测试节点）的网络套接字标识 IP
对于路由器的内部 NAT 局域网策略而言，客户端握手时不能使用类似于 `localhost` 或者闭环测试地址 `127.0.0.1` 这种内部请求路由，其需要确切明确设备的局域网 IPv4。
1. 在运行 FVM 项目的 Windows 机上使用快捷键 `Win + R` 调出命令运行框。
2. 键入 `cmd` 并执行确认打开终端控制器。
3. 键入诊断指令 `ipconfig`，并寻找活动连接所在列表接口（通常表现为 `以太网适配器 Ethernet` 或是局域无线链接的 `无线局域网适配器 WLAN` 等名称）。
4. 找到类似 `IPv4 地址 . . . . . . . . . . . . : 192.168.x.x` 或 `10.x.x.x` 的信息。
5. **请记录该地址**，此地址便是这台高性能服务器的访问坐标点。

### 3.2 编译、运行并且进行防火墙越权
1. 采用你在 VS/CMake 中既有的命令规则执行本应用的二次增量构建。（`web_server.cpp` 已自动被脚本打包在 `PROJECT_CPP_SOURCES`）。宏判断在包含 `ws2_32` 提供 `WinSock` 运行时支撑。
2. 执行或运行编译后的独立程序，请关注其后台打印台输出日志：
    ```
    [Web] 服务器监听开始: http://0.0.0.0:18080
    ```
    （代表其使用 `INADDR_ANY` 特权绑定了全部网络虚拟或真实通讯接口）
3. **关键安全确认**：初次监听任何应用端口均会触发 Windows Defender（或者是部分第三方网络隔离保护系统）的安全防火墙阻塞警报弹出提示框；此时你需要将其网络环境的选区中的 `专用网络（例如家庭或工作场所网络）` 和（或） `公用网络` 悉数选中并点击**允许访问/通行**。若遗漏这一步，任何来源于局域网请求向该 18080 端口握手的数据封包都会遭系统内核级的拦截与丢弃。

### 3.3 Mac 端验证与流式操作交互
保证两台测试终端接入的路由器是同样的 IP 划分配额局域网环境（在同一基站的同一网络环境内）。
1. 在 Mac 终端系统打开现代基准的浏览器应用（通常任意选择 Safari、Google Chrome 皆可保证支持与解析成功）。
2. 在核心网址导航输入访问全路径 `http://<此处替换在 3.1 中拿到的记录 IPv4 地址>:18080` (如不加端口号可能错误解析为 `80` 主配置区)，并按下键车。
3. 返回解析过程发生：
   - 迅速收悉根级 HTML5。
   - `WebSocket` 对象创建抛去 HTTP Upgrade 到服务端请求并获得响应。
4. 如果构建和监听未出差错，浏览器页面中将会毫无违和感地开始刷动着由服务物理端发送压入而成的极其流利和连续不断的渲染数据图像流。
5. 交互性检测：请使用鼠标或者触控板对图形区域内部所展示的原生 `ImGui` 面板上任意复选框和横向调参滚动条实施操作拖拽，并观察模拟场景内的马赫数波、数据统计时间或障碍物流是否相应生变或更新。如果在 Mac 前端的每一个行为立刻呈现，此全栈重构验证即可宣告通项。

## 四、 跨公网网络拓展方案建议
当你未来需要脱开局域网环境进行广域网接入处理时（如 MacBook 置身于公共环境，需要拨接私宅内的计算设备），因 NAT 设备存在不再可通过该内部网段 IPv4 进行点对点直接定址请求：

*   建议采用各类 **SD-WAN 异地虚拟局域网通讯**。例如开源服务 `Tailscale`（底层由 WireGuard 安全传输支撑）、`ZeroTier`（以应用路由管理打通连接点）等产品。在 Windows 后端和服务调用的前端各配置相同的授权密钥或账户实例接入该控制器内，便能在系统内自动添加包含统一虚拟基址的加密网卡接口（可能通常以 `100.x.x.x` 或特定内网分配下发）。之后仅需直接以该 IP 进行原部署指导流程替换，即达到完美解决外网握手的架构局限与内网阻绝难题。
