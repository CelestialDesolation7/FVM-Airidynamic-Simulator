# 串流功能全生命周期数据流

> **前置假设**：读者已完整理解仿真程序的主循环（GL 线程渲染、求解器线程推进、双缓冲 PBO/VBO）。  
> 本文聚焦于**串流子系统**，追踪从服务器启动到最终关闭的完整数据流，以及四种关键场景的详细执行路径。

## 一、串流子系统启动（`StreamServer::initialize`）

串流子系统的启动发生在 `main.cpp` 的 GL 上下文已创建、GLFW 窗口已显示之后，求解器线程启动之前：

```cpp
// main.cpp（启用 ENABLE_STREAMING 宏时）
StreamServer::Config cfg;
cfg.httpPort    = 8080;
cfg.bitrateMbps = 15;
cfg.fps         = 60;
cfg.webRoot     = "Release/web";            // 静态网页文件目录
g_streamServer.initialize(cfg, windowWidth, windowHeight);
```

`initialize()` 按顺序完成以下操作：

### 1.1 获取 CUDA 上下文

```cpp
// stream_server.cpp → initialize()
cuCtxGetCurrent(&cuCtx_);   // 取出当前 GL 线程绑定的 CUDA 上下文
```

此时程序已完成 `gladLoadGLLoader()` 和 `cudaSetDevice(0)`，GL 线程上有一个活跃的 CUDA 主上下文。  
`cuCtx_` 将被传给 `NvencEncoder`，使编码器与渲染使用同一个 GPU 上下文，**数据可在同一 GPU 内存空间无拷贝流转**。

### 1.2 初始化帧捕获器（`FrameCapture::initialize`）

```
FrameCapture::initialize(captureW, captureH)
  ├─ createFBO(0, w, h)
  │    ├─ glGenTextures + glTexImage2D(GL_RGBA8, w, h)  // 创建 RGBA8 纹理
  │    ├─ glGenFramebuffers + glFramebufferTexture2D     // 纹理挂载为颜色附件
  │    ├─ cudaGraphicsGLRegisterImage(texture_[0], ReadOnly)
  │    │      // 向 CUDA 注册该纹理，使 CUDA 可在映射期间读取其内容
  │    └─ cudaMalloc(stagingBuffer_[0], w*h*4)
  │          // 分配线性 CUDA 设备内存，供编码线程无需 GL 上下文即可访问
  └─ createFBO(1, w, h)  // 完全对称的第二套缓冲
```

双缓冲机制：
- `fbo_[0]` / `fbo_[1]`：两个独立的 OpenGL FBO，`writeIndex_` 指示当前写入哪个
- `stagingBuffer_[0]` / `stagingBuffer_[1]`：对应的线性 CUDA 设备内存
- GL 线程写 `fbo_[writeIndex_]` → CUDA 拷贝 → `stagingBuffer_[writeIndex_]`
- `getDevicePtr()` 返回刚写完的那个，并将 `writeIndex_` 翻转
- 编码线程读 `stagingBuffer_[readIdx]`，GL 线程写 `stagingBuffer_[1-readIdx]`，**互不干扰**

### 1.3 初始化 NVENC 编码器（`NvencEncoder::initialize`）

```
NvencEncoder::initialize(cuCtx_, {w, h, 60fps, 15Mbps})
  ├─ loadNvEncApi(fn_)
  │      // 动态加载 nvEncodeAPI64.dll，填充函数指针表 fn_
  │      // NVENC 的所有调用均通过此函数指针表进行，而非直接链接
  ├─ nvEncOpenEncodeSessionEx → encoder_
  │      // 在 CUDA 上下文上打开一个编码会话句柄
  ├─ nvEncGetEncodePresetConfigEx(P3, ULTRA_LOW_LATENCY) → presetConfig
  │      // 取出预设配置，再在此基础上覆盖关键参数：
  │      //   profileGUID  = BASELINE （无 CABAC，兼容性最广）
  │      //   gopLength    = 120       （每 2 秒一个 IDR）
  │      //   frameIntervalP = 1       （禁止 B 帧，P 帧直接引用前一帧）
  │      //   rateControlMode = CBR    （恒定码率，VBV 缓冲 = 1 帧码量）
  │      //   zeroReorderDelay = 1     （编码完立即输出，不等后续帧）
  │      //   repeatSPSPPS = 1         （每个 IDR 前附加 SPS/PPS，使断帧后可恢复）
  ├─ nvEncInitializeEncoder(initParams) → 编码器就绪
  ├─ nvEncCreateBitstreamBuffer → bitstreamBuffer_
  │      // 编码器将 NAL 数据写入此 CPU 侧输出缓冲（通过 Lock/Unlock 访问）
  ├─ cudaMalloc(stagingBuffer_, w*h*4)
  │      // NVENC 的输入暂存缓冲：将来源数据拷到此处再送入 NVENC
  └─ nvEncRegisterResource(stagingBuffer_, ABGR)
         // 告知 NVENC：以后编码时从这个指针取像素，格式 ABGR
         // （与 OpenGL 输出的 RGBA 字节序一致，名字差异来自大小端约定）
```

### 1.4 启动 HTTP 服务器线程（`startHttpServer`）

在独立线程中运行 `httplib::Server::listen("0.0.0.0", 8080)`，注册以下路由：

| 路由 | 方法 | 用途 |
|------|------|------|
| `/` | GET | 静态文件（`index.html`, `stream.js`） |
| `/api/health` | GET | 心跳探测 |
| `/api/offer` | POST | **WebRTC 信令入口**，阻塞直到 ICE 收集完成 |
| `/api/diag` | GET | 调试信息（帧数、字节数、分辨率） |
| `/api/browser-stats` | POST/GET | 浏览器上报/查询 WebRTC 统计 |

### 1.5 启动编码线程（`encodeLoop`）

```cpp
encodeThread_ = std::thread(&StreamServer::encodeLoop, this);
```

编码线程立即进入等待状态：

```cpp
// encodeLoop 开头
cuCtxSetCurrent(cuCtx_);  // 绑定同一个 CUDA 上下文（NVENC API 要求）

while (running_) {
    // 阻塞等待 GL 线程发出"有新帧"信号
    encodeCV_.wait(lock, [&] { return frameAvailable_ || !running_; });
    ...
}
```

**至此，串流子系统已完全就绪，等待浏览器连接。**

## 二、建立第一条 WebRTC 连接

### 2.1 浏览器端发起连接（`stream.js → connect()`）

用户在浏览器中打开 `http://<服务器IP>:8080`，加载 `index.html`，`stream.js` 立即调用 `connect()`：

```javascript
// stream.js → connect()

// ① 创建 RTCPeerConnection，无 ICE 服务器（假设局域网直连，无需 STUN/TURN）
pc = new RTCPeerConnection({ iceServers: [] });

// ② 声明接收视频（服务器推流，浏览器只收）
pc.addTransceiver("video", { direction: "recvonly" });

// ③ 创建 DataChannel（有序，用于向服务器回传输入事件）
dc = pc.createDataChannel("input", { ordered: true });
dc.onopen = () => sendResize();  // DataChannel 就绪后立即发送当前窗口尺寸

// ④ 生成 Offer SDP（浏览器宣告自己支持的编解码器列表）
const offer = await pc.createOffer();
await pc.setLocalDescription(offer);

// ⑤ 等待 ICE 候选收集完成（最多 3 秒），确保 SDP 中包含网络地址
await waitGathering(pc, 3000);

// ⑥ 将 Offer SDP 发送给服务器，等待 Answer（带 5 秒超时）
const controller = new AbortController();
const fetchTimeout = setTimeout(() => controller.abort(), 5000);
const resp = await fetch("/api/offer", {
    method: "POST",
    body: pc.localDescription.sdp,
    signal: controller.signal,
});
clearTimeout(fetchTimeout);

// ⑦ 将服务器返回的 Answer SDP 设为远端描述，连接开始协商
await pc.setRemoteDescription({ type: "answer", sdp: await resp.text() });
```

### 2.2 服务器接收 Offer（`/api/offer` 处理器）

`/api/offer` 在 httplib 的线程池中执行：

```cpp
// stream_server.cpp → startHttpServer() 内的 lambda

// ① 获取序列化锁：同一时刻只允许一个 Offer 在处理
//    若有第二个浏览器窗口同时请求，它将在此阻塞等待
std::lock_guard<std::mutex> offerLk(offerMutex_);

// ② 创建局部同步原语，用于等待 ICE 候选收集完成
std::mutex mtx;
std::condition_variable cv;
bool gatheringDone = false;

// ③ 调用核心函数，建立 PeerConnection、视频 Track、DataChannel
createPeerConnection(req.body, mtx, cv, gatheringDone);

// ④ 阻塞等待 ICE 候选收集完成（最多 5 秒）
//    收集完成后 Answer SDP 才包含服务器的 IP/端口
cv.wait_for(lk, std::chrono::seconds(5), [&] { return gatheringDone; });

// ⑤ 将完整的 Answer SDP 作为 HTTP 响应体返回给浏览器
res.set_content(pc_->localDescription()->generateSdp(), "application/sdp");
```

### 2.3 `createPeerConnection` 详细流程

这是连接建立的核心函数，执行以下步骤：

**步骤 A：拆除旧连接（若存在）**

```cpp
if (pc_) {
    hasClient_ = false;          // 立即停止 onFrameReady() 的捕获触发
    pc_->close();                // 关闭旧 DTLS/ICE 连接
    pc_.reset();                 // 释放旧 PeerConnection 对象

    std::lock_guard<std::mutex> lk(sendMutex_);
    videoTrack_.reset();         // 在 sendMutex_ 保护下清空 Track 指针
    rtpConfig_.reset();          // 使编码线程回调中的本地副本失效
    // （编码线程已持有的局部 shared_ptr 此时引用计数不降为 0，
    //   待本次编码完成后自然析构，不会崩溃）
}
```

**步骤 B：创建新的 `rtc::PeerConnection`**

```cpp
rtc::Configuration rtcConfig;  // 空配置 = 无 ICE 服务器，仅局域网直连
pc_ = std::make_shared<rtc::PeerConnection>(rtcConfig);
```

**步骤 C：注册回调**

```cpp
// ICE 候选收集完成回调（通知 /api/offer 处理器可以返回 Answer 了）
pc_->onGatheringStateChange([&gatherMtx, &gatherCV, &gatheringDone](auto state) {
    if (state == GatheringState::Complete) {
        gatheringDone = true;
        gatherCV.notify_one();
    }
});

// 连接状态机回调（在 libdatachannel 内部线程触发）
pc_->onStateChange([this](rtc::PeerConnection::State state) {
    if (state == State::Connected) {
        hasClient_ = true;                  // 开启帧捕获触发
        streamStartTime_ = steady_clock::now(); // 重置 RTP 时间戳基准
        encoder_.forceKeyframe();           // 连接就绪时立即发一个 IDR 帧
    } else if (state == Disconnected || state == Failed || state == Closed) {
        hasClient_ = false;                 // 关闭帧捕获触发
    }
});
```

**步骤 D：解析 Offer SDP，协商 H.264 参数**

浏览器的 Offer SDP 的视频段可能包含多个 H.264 配置，例如：

```sdp
m=video 9 UDP/TLS/RTP/SAVPF 97 109 98 35
a=rtpmap:97 H264/90000
a=fmtp:97 level-asymmetry-allowed=1;packetization-mode=1;profile-level-id=42e01f
a=rtpmap:109 H264/90000
a=fmtp:109 level-asymmetry-allowed=1;packetization-mode=0;profile-level-id=42001f
a=mid:0
```

服务器的处理流程：

```cpp
// 提取 a=mid 值（视频媒体流标识符，用于多路复用时区分流）
std::string videoMid = extractMid(offerSdp, "video");  // → "0"

// 解析所有 H264 条目，按评分选最优
auto allH264 = findAllH264(offerSdp);   // 找到 PT=97 和 PT=109
auto chosen  = pickBestH264(allH264);
// PT=97 有 packetization-mode=1（+100）且 profile=42e0（+50），总分 150
// PT=109 无 packetization-mode=1，总分 0
// → 选择 PT=97，fmtp="level-asymmetry-allowed=1;packetization-mode=1;profile-level-id=42e01f"
```

`packetization-mode=1` 是关键：它允许将超过 MTU（~1200 字节）的 NALU 切分为多个 FU-A RTP 包，与 NVENC 实际输出的较大 NALU 格式完全匹配。

**步骤 E：构建视频 Track 和 RTP 媒体处理链**

```cpp
// 创建 SendOnly 视频 Track（服务器只发，不收视频）
rtc::Description::Video media(videoMid, SendOnly);
media.addH264Codec(97, "...fmtp...");   // 使用协商出的 PT 和 fmtp
media.addSSRC(rtpSsrc_, "video-stream"); // 标识该 Track 的 SSRC（同步源标识符）
auto newTrack = pc_->addTrack(media);

// 构建责任链（每层只处理自己关心的信号，其余透传）：
//
//  [NVENC NAL 数据]
//       ↓
//  H264RtpPacketizer   ← 将 NALU 按 MTU 切分为 RTP 包，加序列号和时间戳
//       ↓
//  RtcpSrReporter      ← 定期生成 RTCP SR 报文，让浏览器校准播放时钟
//       ↓
//  RtcpNackResponder   ← 缓存已发 RTP 包；收到浏览器 NACK 时重发丢失包
//       ↓
//  PliHandler          ← 收到浏览器 PLI（画面解码失败）时调用 forceKeyframe()
//       ↓
//  [发往浏览器 UDP 包]

auto newRtpConfig = std::make_shared<rtc::RtpPacketizationConfig>(
    rtpSsrc_, "video-stream", uint8_t(h264pt),
    H264RtpPacketizer::ClockRate);  // 90000 Hz，H.264 标准时钟频率

auto packetizer = std::make_shared<H264RtpPacketizer>(
    Separator::StartSequence, newRtpConfig);
// Separator::StartSequence：以 H.264 起始码（0x00 00 00 01）为 NALU 分隔符

auto srReporter   = std::make_shared<RtcpSrReporter>(newRtpConfig);
auto nackResponder = std::make_shared<RtcpNackResponder>();
auto pliHandler    = std::make_shared<PliHandler>([this]() { encoder_.forceKeyframe(); });

packetizer->addToChain(srReporter);
srReporter->addToChain(nackResponder);
nackResponder->addToChain(pliHandler);
newTrack->setMediaHandler(packetizer);
```

**步骤 F：原子发布 Track 和 RtpConfig**

```cpp
// 必须在 sendMutex_ 下同时更新两者，确保编码线程永远看到配对一致的指针
// （不会出现"新 track + 旧 rtp" 或 "旧 track + 新 rtp" 的中间状态）
std::lock_guard<std::mutex> lk(sendMutex_);
videoTrack_ = newTrack;
rtpConfig_  = newRtpConfig;
```

**步骤 G：注册 DataChannel 回调并设置远端描述**

```cpp
pc_->onDataChannel([this](auto dc) {
    dataChannel_ = dc;
    // 浏览器每次输入操作都通过此 DataChannel 发 JSON 字符串
    dc->onMessage([this](auto msg) {
        inputHandler_.onMessage(std::get<std::string>(msg));  // 异步入队
    });
});

// 触发信令协商：libdatachannel 内部处理 Offer → 生成 Answer → 收集 ICE 候选
pc_->setRemoteDescription(Description(offerSdp, Type::Offer));
// ↑ 此调用返回后，ICE 候选收集在内部线程异步进行
//   收集完成时触发 onGatheringStateChange → gatheringDone = true → cv.notify_one()
```

### 2.4 ICE 候选收集与连接建立

ICE（Interactive Connectivity Establishment）是 WebRTC 用于在 NAT 后发现实际可用网络路径的机制：

```
服务器（libdatachannel）                     浏览器
  │                                            │
  │ 收集本地网络接口地址（host candidates）     │
  │ ← 如 192.168.1.100:xxxxx                   │
  │                                            │
  │← Answer SDP（含 ICE 候选列表）── HTTP 响应 │
  │                                            │
  │══════════ DTLS 握手（加密协商）════════════│
  │                                            │
  │ onStateChange → Connected                  │
  │   hasClient_ = true                        │
  │   encoder_.forceKeyframe()                 │
  │   streamStartTime_ = now()                 │
```

对于局域网场景，双方都在同一网段，host candidates 直接可达，无需 STUN/TURN。

## 三、正常传输画面（稳定串流阶段）

连接建立后，每当 GL 线程完成一帧渲染，就触发一次完整的捕获-编码-发送流水线。

### 3.1 GL 线程触发帧捕获（`onFrameReady`）

`onFrameReady()` 由 `main.cpp` 主循环在 `glfwSwapBuffers()` 之后调用：

```cpp
// main.cpp 主循环（每帧）
renderer.render(params);         // OpenGL 绘制完成，结果在默认帧缓冲中
ImGui::Render();
ImGui_ImplOpenGL3_RenderDrawData(ImGui::GetDrawData());
glfwSwapBuffers(window);         // 交换前后缓冲，画面显示到屏幕

#ifdef ENABLE_STREAMING
g_streamServer.onFrameReady();   // ← 触发串流捕获
#endif
```

`onFrameReady()` 的执行路径：

```cpp
void StreamServer::onFrameReady() {
    // ① 守门检查：串流未运行或无客户端连接时直接返回，不产生任何开销
    if (!running_ || !hasClient_) return;

    // ② 检查是否有待应用的分辨率变化（见第五章）
    applyPendingResize();

    // ③ 执行帧捕获（在 GL 线程上，持有 GL 上下文）
    capture_.capture();

    // ④ 设置帧就绪标志并唤醒编码线程
    {
        std::lock_guard<std::mutex> lock(encodeMutex_);
        frameAvailable_ = true;
    }
    encodeCV_.notify_one();
    // GL 线程到此返回，继续下一帧渲染，编码由独立线程异步完成
}
```

### 3.2 帧捕获细节（`FrameCapture::capture`）

`capture()` 在 GL 线程上执行，完成从 OpenGL 帧缓冲到 CUDA 线性内存的完整搬运：

```
glfwSwapBuffers 之后，前台缓冲已包含刚渲染完的仿真画面
        │
        ▼
glBlitFramebuffer(                         // 第一步：GPU 内部 Blit
    src: 0, 0 → width_, height_,           //   来源：默认帧缓冲（前台）
    dst: 0, height_ → width_, 0,           //   目标：FBO[writeIndex_] 附带的 RGBA8 纹理
    COLOR_BUFFER_BIT, GL_LINEAR)           //   Y 轴翻转：补偿 OpenGL 坐标原点在左下角
        │                                  //   （若不翻转，浏览器看到的画面会上下颠倒）
        ▼
cudaGraphicsMapResources(cudaResource_[writeIndex_])
        // 向 CUDA 声明：此纹理现在交给 CUDA 访问，OpenGL 不得修改
cudaGraphicsSubResourceGetMappedArray(→ array)
        // 获取纹理对应的 CUDA 数组句柄
cudaMemcpy2DFromArray(                     // 第二步：CUDA 内部 2D 拷贝
    dst: stagingBuffer_[writeIndex_],      //   目标：线性内存（行步长 = width*4）
    src: array,                            //   来源：CUDA 数组（二维网格布局）
    cudaMemcpyDeviceToDevice)              //   全程在 GPU 显存内完成，不经过 CPU
cudaGraphicsUnmapResources(...)
        // 归还控制权给 OpenGL（此后 OpenGL 可重新渲染到该 FBO）
```

**为什么需要从 CUDA 数组拷贝到线性内存？**  
NVENC 要求输入为"线性（linear）"布局——即行数据连续排列、行步长固定。CUDA 数组（`cudaArray`）使用硬件优化的 swizzle 平铺布局，虽然适合纹理采样，但不能直接传给 NVENC。`cudaMemcpy2DFromArray` 完成这一格式转换，代价仅为 GPU 内部的带宽消耗，约 1-2 ms（1920×1080 约 8MB）。

### 3.3 编码线程处理（`encodeLoop`）

编码线程被 `encodeCV_.notify_one()` 唤醒后：

```cpp
void StreamServer::encodeLoop() {
    cuCtxSetCurrent(cuCtx_);   // 在编码线程上也绑定同一 CUDA 上下文

    while (running_) {
        // ① 等待帧就绪信号（条件变量，零 CPU 占用地等待）
        {
            std::unique_lock<std::mutex> lock(encodeMutex_);
            encodeCV_.wait(lock, [&] { return frameAvailable_ || !running_; });
            if (!running_) break;
            frameAvailable_ = false;  // 消费信号
        }

        // ② 尝试获取流水线锁（非阻塞 try_lock）
        //    如果 GL 线程正在执行 applyPendingResize（需要重建 FBO/编码器），
        //    则跳过本帧编码，避免两者同时访问捕获器和编码器造成竞争
        std::unique_lock<std::mutex> pipelineLock(pipelineMutex_, std::try_to_lock);
        if (!pipelineLock.owns_lock()) continue;  // resize 中，丢弃此帧，等下一帧

        // ③ 从捕获器取出设备指针（同时翻转双缓冲写索引）
        void *devPtr = capture_.getDevicePtr();
        // 现在：devPtr 指向 stagingBuffer_[readIdx]（刚 capture() 写完的那个）
        //       GL 线程下次 capture() 将写 stagingBuffer_[1-readIdx]

        // ④ 编码一帧（见 3.4 节），通过 lambda 回调接收编码结果
        encoder_.encodeFrame(devPtr, capture_.getPitch(), [this](...) {
            // 在 sendMutex_ 下复制 shared_ptr（避免信令线程同时替换全局指针）
            std::shared_ptr<rtc::Track> track;
            std::shared_ptr<rtc::RtpPacketizationConfig> rtp;
            {
                std::lock_guard<std::mutex> lk(sendMutex_);
                track = videoTrack_;
                rtp   = rtpConfig_;
            }
            if (!track || !track->isOpen() || !rtp) return;

            // ⑤ 计算 RTP 时间戳
            //    H.264 时钟频率 = 90000 Hz
            //    timestamp = 起始值 + 距会话开始的微秒数 × (90000/1000000)
            auto elapsed = duration_cast<microseconds>(now - streamStartTime_).count();
            rtp->timestamp = rtp->startTimestamp + uint32_t(elapsed * 90000ULL / 1000000ULL);

            // ⑥ 发送 RTP 数据（libdatachannel 内部经过 H264RtpPacketizer 分片后发出）
            track->send(reinterpret_cast<const std::byte*>(data), size);
        });
    }
}
```

### 3.4 NVENC 编码一帧（`NvencEncoder::encodeFrame`）

```
devPtr（FrameCapture 暂存缓冲，线性 RGBA）
        │
        ▼
cuMemcpy2D(src=devPtr, dst=stagingBuffer_)
        // 从 FrameCapture 的暂存缓冲拷到 NvencEncoder 自己的暂存缓冲
        // （两个 staging 都在 GPU，这步是 GPU→GPU，约 0.1ms）
        // 为什么要两个 staging？
        //   FrameCapture 的 staging 是自己管理的，不在 NVENC 注册范围内；
        //   NVENC 只接受通过 nvEncRegisterResource 注册过的内存指针
        │
        ▼
nvEncMapInputResource(registeredRes_) → mapRes.mappedResource
        // 将 NVENC 注册资源映射为此次编码的输入句柄
        │
        ▼
nvEncEncodePicture(picParams)
        // picParams 关键字段：
        //   inputBuffer  = mappedResource（本帧像素数据）
        //   outputBitstream = bitstreamBuffer_（NAL 输出目的地）
        //   pictureStruct = FRAME（逐行扫描帧，非场）
        //   encodePicFlags |= FORCEIDR（若 forceIdr_ == true）
        // 超低延迟模式下：NVENC 编码完立即输出，不等后续帧（zeroReorderDelay=1）
        │
        ▼
nvEncUnmapInputResource(mappedResource)
        │
        ▼
processOutput(cb):
    nvEncLockBitstream(bitstreamBuffer_) → lockBs
        // 锁定输出缓冲，获取 NAL 数据指针和大小
    isKey = (pictureType == IDR || pictureType == I)
    cb(lockBs.bitstreamBufferPtr, lockBs.bitstreamSizeInBytes, isKey)
        // ← 在此处触发 encodeLoop 中的 lambda，计算时间戳并发送 RTP
    nvEncUnlockBitstream(bitstreamBuffer_)
```

### 3.5 RTP 打包与发送（libdatachannel 内部）

`track->send(data, size)` 调用后，数据经过媒体处理链：

```
H264RtpPacketizer::send(nalu_data, size)
    │
    ├─ 若 size > MTU（~1200 字节）：切分为 FU-A 分片
    │    FU-A 头部：[F|NRI|28] [S|E|R|nal_unit_type] [payload...]
    │    第一片 S=1，最后片 E=1，其余 S=E=0
    │
    ├─ 若 size ≤ MTU：单包（Single NALU）
    │
    └─ 每个 RTP 包：
         [12字节 RTP 头][payload]
         RTP 头含：序列号（单调递增）、时间戳（步骤 ⑤ 计算的值）、SSRC
    │
    ▼
RtcpSrReporter（透传 RTP 包，定期生成 SR 报文）
    ▼
RtcpNackResponder（缓存 RTP 包，监听 RTCP NACK）
    ▼
PliHandler（监听 RTCP PLI）
    ▼
libdatachannel DTLS 层（加密）
    ▼
UDP 发往浏览器
```

浏览器收到 RTP 包后，WebRTC 内置解码器（通常是 ExternalDecoder，调用 GPU 硬解）完成 H.264 解码，渲染到 `<video>` 元素。

## 四、浏览器输入回传

### 4.1 DataChannel 就绪

WebRTC 连接建立后，浏览器端的 DataChannel 会触发 `onopen`：

```javascript
// stream.js
dc.onopen = () => {
    sendResize();  // ← DataChannel 一开放立即发送当前窗口尺寸
};
```

服务器端，`createPeerConnection` 中注册了 DataChannel 回调：

```cpp
pc_->onDataChannel([this](auto dc) {
    dataChannel_ = dc;
    dc->onMessage([this](auto msg) {
        inputHandler_.onMessage(std::get<std::string>(msg));  // 异步解析入队
    });
});
```

`dc->onMessage` 在 **libdatachannel 的内部网络线程**中触发，与 GL 线程并发执行。`inputHandler_.onMessage()` 通过 `mutex_` 保护队列写入，GL 线程后续通过 `processEvents()` 消费。

### 4.2 输入事件的完整路径

以用户在浏览器中移动鼠标为例：

```
浏览器页面（stream.js）
  video.addEventListener("mousemove", (e) => {
      const r = video.getBoundingClientRect();
      // 将鼠标坐标归一化为 [0,1]，消除分辨率差异
      send({ type: "mousemove",
             x: (e.clientX - r.left) / r.width,
             y: (e.clientY - r.top)  / r.height });
  });
  dc.send(JSON.stringify(obj))  ← DataChannel 发送 JSON 字符串
        │
        ▼ （UDP + DTLS 传输）
        │
服务器 libdatachannel 网络线程
  dc->onMessage 回调触发
  inputHandler_.onMessage('{"type":"mousemove","x":0.5,"y":0.3}')
        │
  InputHandler::onMessage() 内部：
    type = jsonStr(json, "type")  → "mousemove"
    ev.x = jsonNum(json, "x")    → 0.5
    ev.y = jsonNum(json, "y")    → 0.3
    queue_.push(ev)               ← 加入事件队列（持锁）
        │
        ▼ （下一帧 GL 线程安全区域）
        │
main.cpp 主循环
  g_streamServer.inputHandler().processEvents(window, winW, winH, onResize)
        │
  InputHandler::processEvents():
    std::swap(snapshot, queue_)   ← 一次性取出全部事件（持锁，时间极短）
    // 之后在无锁状态下处理 snapshot，避免长时间持锁阻塞网络线程
        │
  for (Event ev : snapshot):
    case MOUSE_MOVE:
      double px = ev.x * winW;  // 反归一化：[0,1] → 像素坐标
      double py = ev.y * winH;
      io.AddMousePosEvent(px, py);  ← 注入 ImGui 鼠标位置
      // ImGui 后端自动更新 GLFW 鼠标状态，使 UI 响应
```

### 4.3 键盘事件的双路注入

键盘事件同时需要注入 ImGui（UI 响应）和触发 GLFW 键盘回调（`keyCallback` 处理 ESC/Space/R 等快捷键）：

```cpp
case Event::KEY_DOWN:
    if (ev.key != GLFW_KEY_UNKNOWN) {

        auto keyCb = glfwSetKeyCallback(window, nullptr);

        //   ESC   → glfwSetWindowShouldClose(window, true)
        //   Space → params.paused = !params.paused
        //   R     → g_resetRequested = true
    }
```

## 五、浏览器窗口尺寸变化（分辨率同步）

### 5.1 浏览器端检测尺寸变化

```javascript
// stream.js
window.addEventListener("resize", () => {
    // 防抖：300ms 内连续触发只执行最后一次
    clearTimeout(window._resizeDebounce);
    window._resizeDebounce = setTimeout(sendResize, 300);
});

function sendResize() {
    const dpr = window.devicePixelRatio || 1;  // HiDPI 缩放因子（Retina 屏为 2.0）
    const cssW = window.visualViewport?.width ?? window.innerWidth;
    const cssH = window.visualViewport?.height ?? window.innerHeight;

    // 转换为物理像素，并对齐到偶数（H.264 YUV 4:2:0 要求宽高均为偶数）
    const w = Math.max(64, Math.round(cssW * dpr) & ~1);
    const h = Math.max(64, Math.round(cssH * dpr) & ~1);

    if (w === lastResizeW && h === lastResizeH) return;  // 去重：尺寸未变则跳过
    lastResizeW = w; lastResizeH = h;
    send({ type: "resize", width: w, height: h });  // 通过 DataChannel 发送
}
```

DataChannel 建立时也会立即调用 `sendResize()`，确保服务器从第一帧起就以正确分辨率编码。

### 5.2 服务器端异步应用尺寸变化

Resize 信号的传递路径：

```
DataChannel 消息 → InputHandler::onMessage
    ev.type = RESIZE, ev.x = w, ev.y = h
    queue_.push(ev)
    pendingResizeW_ = w; pendingResizeH_ = h; resizePending_ = true
        │
        ▼ （下一帧 GL 线程安全区域）
        │
processEvents() → onResize 回调（在 main.cpp 中注册）
    → StreamServer::requestResize(w, h)
        pendingW_.store(w);   // 原子写，立即返回，不阻塞 GL 线程
        pendingH_.store(h);
        │
        ▼ （下一帧 onFrameReady 开头）
        │
StreamServer::applyPendingResize()
    int w = pendingW_.load(), h = pendingH_.load();
    if (w == capture_.getWidth() && h == capture_.getHeight()) return;  // 尺寸未变

    // 尝试非阻塞获取流水线锁
    // 若编码线程正在 encodeFrame（持有 pipelineMutex_），则跳过本帧，下帧再试
    // 这确保 GL 线程永不被编码线程阻塞
    std::unique_lock<std::mutex> lock(pipelineMutex_, std::try_to_lock);
    if (!lock.owns_lock()) return;

    // 执行实际 resize：销毁旧 FBO+staging，创建新尺寸的 FBO+staging
    capture_.resize(w, h);
    // 重建编码器（NVENC 不支持热修改分辨率，必须完整重初始化）
    encoder_.cleanup();
    encoder_.initialize(cuCtx_, {w, h, fps, bitrate});
    // 强制下一帧为 IDR，使浏览器在新分辨率下可立即解码
    encoder_.forceKeyframe();
```

**为什么用两级原子存储而不是直接在回调里 resize？**

DataChannel 回调在 libdatachannel 内部线程触发，该线程不持有 GL 上下文（OpenGL 调用会崩溃）。`requestResize` 只做原子写，真正的 resize 延迟到 GL 线程的 `onFrameReady` 中执行，安全且无阻塞。

## 六、切换连接（新浏览器窗口踢旧窗口）

当第二个浏览器窗口访问 `http://服务器:8080` 时，`stream.js` 同样调用 `connect()`，向 `/api/offer` 发送一个新的 Offer。

### 6.1 `offerMutex_` 的序列化效果

```
时间轴：

窗口 A 已建立连接，正在串流
                                    窗口 B 打开，发送 POST /api/offer
                                          │
                              httplib 线程池线程 B 进入处理器
                              → lock_guard<mutex> offerLk(offerMutex_)
                              → 获锁成功（此时无其他 offer 在处理）
                              → createPeerConnection(offerSdpB, ...)
                                    │
                                    ├─ pc_ 非空（窗口 A 的连接）
                                    │   hasClient_ = false  ← 停止向窗口 A 发帧
                                    │   pc_->close()        ← 关闭 A 的 DTLS 连接
                                    │   pc_.reset()
                                    │   videoTrack_.reset() ← 在 sendMutex_ 下
                                    │   rtpConfig_.reset()
                                    │   dataChannel_.reset()
                                    │
                                    └─ 创建新 pc_（窗口 B 的连接）
                                       注册回调，构建 Track/Handler chain
                                       发布新 videoTrack_/rtpConfig_（sendMutex_）
                                       setRemoteDescription(offerSdpB)
                              │
                              等待 ICE 收集（gatheringDone）
                              返回 Answer SDP → 窗口 B 建立连接

窗口 A 收到 ICE disconnect（其 PeerConnection 已被关闭）
→ oniceconnectionstatechange: "disconnected"
→ scheduleReconnect()（3 秒后重连）
→ 3 秒后 connect() → POST /api/offer
  → 此时 offerMutex_ 空闲，获锁
  → createPeerConnection 又会踢掉窗口 B
  → 如此往复，最后访问的窗口保持连接
```

若两个窗口同时发送 Offer（`offerMutex_` 确保它们串行处理）：先到的那个先建立连接，后到的那个进入后会拆除先建立的连接，最终后来者获得连接。

## 七、连接中断与重连

### 7.1 中断检测

连接中断（网络波动、浏览器刷新、服务器端主动关闭）通过两个机制检测：

**服务器端**（libdatachannel 内部线程触发 `onStateChange`）：

```cpp
pc_->onStateChange([this](State state) {
    if (state == Disconnected || state == Failed || state == Closed) {
        hasClient_ = false;  // GL 线程的 onFrameReady() 将从下一帧起不再捕获/编码
    }
});
```

**浏览器端**（WebRTC 内置 ICE 超时检测）：

```javascript
pc.oniceconnectionstatechange = () => {
    if (s === "disconnected" || s === "failed" || s === "closed") {
        showStatus("连接断开，正在重连...");
        scheduleReconnect();  // 3 秒后再次调用 connect()
    }
};
```

### 7.2 重连时的关键细节：`lastResizeW/H` 重置

```javascript
function cleanup() {
    if (statsTimer) { clearInterval(statsTimer); statsTimer = null; }
    if (pc) { pc.close(); pc = null; }
    dc = null;
    video.srcObject = null;
    lastResizeW = 0;  // ← 关键：重置为 0
    lastResizeH = 0;  // ← 关键：重置为 0
}
```

若不重置，重连后 DataChannel 就绪时 `sendResize()` 会对比 `lastResizeW/H` 与当前尺寸——若尺寸恰好没有变化，去重逻辑会跳过发送，服务器维持旧分辨率，可能产生黑边（旧连接的分辨率与新连接不一致时）。

重置为 0 后，`sendResize()` 一定会发送，服务器必然执行一次 `applyPendingResize()`，确保新连接的分辨率与浏览器当前尺寸一致。

## 八、程序终止（串流关闭）

### 8.1 触发来源

用户关闭应用窗口（ESC 键或点击叉号），`main.cpp` 主循环退出，调用：

```cpp
#ifdef ENABLE_STREAMING
g_streamServer.shutdown();
#endif
```

### 8.2 `StreamServer::shutdown()` 分步执行

```cpp
void StreamServer::shutdown() {
    if (!running_) return;

    // ① 设置停止标志，并唤醒编码线程使其退出 wait
    running_ = false;
    encodeCV_.notify_all();

    // ② 等待编码线程退出
    //    编码线程检测到 !running_ 后跳出循环，线程函数返回
    if (encodeThread_.joinable()) encodeThread_.join();

    // ③ 关闭 WebRTC 连接（若有）
    //    pc_->close() 触发 DTLS 关闭握手，浏览器收到后进入 "closed" 状态
    //    浏览器将在 3 秒后尝试重连（但服务器已不再响应 /api/offer）
    if (pc_) { pc_->close(); pc_.reset(); }
    videoTrack_.reset();
    dataChannel_.reset();
    rtpConfig_.reset();

    // ④ 停止 HTTP 服务器
    if (httpServer_) httpServer_->stop();
    if (httpThread_.joinable()) httpThread_.join();

    // ⑤ 在持有 pipelineMutex_ 的情况下清理 GPU 资源
    //    此时编码线程已结束，pipelineMutex_ 肯定可以获取
    //    encoder_.cleanup()：注销 NVENC 资源、销毁编码会话、释放 CUDA 内存
    //    capture_.cleanup()：注销 CUDA-GL 互操作资源、删除 FBO/纹理
    {
        std::lock_guard<std::mutex> pipelineLock(pipelineMutex_);
        encoder_.cleanup();
        capture_.cleanup();
    }
}
```

### 8.3 资源释放顺序的约束

串流子系统的资源必须在 GLFW 窗口销毁之前全部释放，因为：
- `capture_.cleanup()` 中调用 `cudaGraphicsUnregisterResource()`，需要 CUDA-GL 互操作上下文仍然有效
- GL 上下文由 GLFW 窗口持有，`glfwDestroyWindow()` 会销毁 GL 上下文

`main.cpp` 中的资源释放顺序为：

```
g_streamServer.shutdown()      ← 先释放串流 GPU 资源（CUDA-GL 互操作注销）
    ↓
renderer.cleanup()             ← 释放仿真渲染 GPU 资源
    ↓
glfwDestroyWindow(window)      ← 最后销毁 GL 上下文
glfwTerminate()
```

在稳定运行阶段，串流子系统的核心状态变量如下：

| 变量 | 类型 | 值域 | 所属线程 |
|------|------|------|---------|

| `dataChannel_` | `shared_ptr` | — | libdatachannel 线程写，DC 消息触发 |
