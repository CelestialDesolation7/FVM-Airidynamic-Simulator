# 浏览器串流功能规划与实现文档

---

## 一、功能概述

### 1.1 目标场景

本模拟器是一个运行在服务端 GPU 上的实时 CFD（计算流体动力学）程序，其渲染结果以 OpenGL 窗口的形式呈现。串流功能的目标是：

> **将 GPU 渲染的 OpenGL 画面以低延迟视频流的方式推送到任意浏览器，同时将浏览器的鼠标、键盘等输入回传到服务器，实现远程交互式控制。**

使用场景包括：
- 在无 GPU 的客户端机器（如笔记本、平板）上远程操控运行在工作站/服务器上的仿真程序
- 通过局域网或公网浏览器访问仿真结果，无需安装任何客户端软件

### 1.2 技术选型理由

| 需求 | 选择 | 理由 |
|------|------|------|
| 视频编码 | **NVENC H.264** | GPU 专用硬件编码单元，编码本身几乎不占 CPU/GPU 渲染算力 |
| 传输协议 | **WebRTC** | 浏览器原生支持，内置可靠/不可靠信道、NAT 穿越、拥塞控制；H.264 over RTP 是其标准 |
| 信令 | **HTTP（httplib）** | SDP 协商仅需一次请求/响应，轻量无状态，无需 WebSocket |
| WebRTC 库 | **libdatachannel** | 纯 C++17，无 GDK/Qt 依赖，同时提供 DataChannel（用于输入回传）和 Track（用于视频） |
| 帧捕获 | **OpenGL FBO + CUDA 互操作** | 零 CPU 拷贝：GPU 渲染结果直接在 GPU 上被 CUDA 取走送入编码器 |

## 二、整体架构

整个串流管道由五个层次构成，数据从左向右单向流动，控制信号（输入事件）从右向左回传：

```
┌──────────────────────────────────────────────────────────────────┐
│  服务端（C++ / CUDA / OpenGL）                                    │
│                                                                  │
│  GL 线程                                                         │
│  ┌──────────────┐    ┌─────────────────────────────────────────┐ │
│  │ OpenGL 渲染  │───▶│         FrameCapture                    │ │
│  │（前台帧缓冲）│    │  FBO Blit + CUDA-GL 互操作 + 双缓冲    │ │
│  └──────────────┘    └────────────────┬────────────────────────┘ │
│                                       │ CUDA 设备指针             │
│  编码线程                              ▼                          │
│                       ┌──────────────────────────────────────┐   │
│                       │         NvencEncoder                │   │
│                       │   NVENC ASIC H.264 硬件编码         │   │
│                       └────────────────┬─────────────────────┘   │
│                                        │ NAL 单元字节流           │
│  httplib 线程池                         ▼                         │
│                       ┌──────────────────────────────────────┐   │
│                       │         StreamServer                │   │
│                       │  libdatachannel WebRTC 打包/发送     │   │
│                       └────────┬─────────────┬───────────────┘   │
│                                │ HTTP 信令    │ DataChannel 输入  │
└────────────────────────────────┼─────────────┼───────────────────┘
                                 │             │
                    HTTP POST    │             │ JSON 事件
                    /api/offer   │             │
                                 ▼             ▼
┌──────────────────────────────────────────────────────────────────┐
│  浏览器（stream.js + index.html）                                 │
│                                                                  │
│  RTCPeerConnection ──▶ <video> 元素（H.264 解码 + 渲染）         │
│  RTCDataChannel    ◀── 鼠标/键盘/窗口尺寸 JSON 消息              │
└──────────────────────────────────────────────────────────────────┘
```

### 2.1 线程模型

| 线程 | 归属 | 职责 |
|------|------|------|
| GL 线程（主线程） | `main.cpp` | OpenGL 渲染、帧捕获（`capture_.capture()`）、输入事件消费 |
| 编码线程 | `StreamServer::encodeLoop()` | 等待帧就绪信号，读取暂存缓冲，调用 NVENC 编码，发送 RTP |
| HTTP 线程 | `httplib::Server::listen()` | 接受浏览器 HTTP 连接，处理 SDP 协商、诊断 API |
| 求解器线程 | `solverThreadFunc()` | CUDA 仿真推进，与主线程通过条件变量同步 |

线程间共享资源及其保护机制：

| 共享资源 | 保护锁 | 读者 | 写者 |
|----------|--------|------|------|
| `videoTrack_` / `rtpConfig_` | `sendMutex_` | 编码线程 | HTTP/信令线程 |
| `FrameCapture` / `NvencEncoder` | `pipelineMutex_`（try_lock） | 编码线程 | GL 线程（resize 时） |
| 帧就绪标志 `frameAvailable_` | `encodeMutex_` + 条件变量 | 编码线程 | GL 线程 |
| `/api/offer` 处理序列化 | `offerMutex_` | — | HTTP 线程（阻塞排队） |
| 浏览器统计数据 | `browserStatsMutex_` | HTTP GET 处理器 | HTTP POST 处理器 |

## 三、帧捕获（`FrameCapture`）

### 3.1 背景知识：OpenGL 帧缓冲与 CUDA 互操作

OpenGL 的渲染结果存放在"帧缓冲对象（FBO）"的颜色附件（`COLOR_ATTACHMENT0`）中，其底层是一张 GPU 纹理（`GL_TEXTURE_2D`）。

CUDA 提供了"CUDA-GL 互操作"机制：通过 `cudaGraphicsGLRegisterImage()` 将 OpenGL 纹理注册为 CUDA 可访问资源，再通过 `cudaGraphicsMapResources()` / `cudaGraphicsSubResourceGetMappedArray()` 在 CUDA 内核中直接读写该纹理——全程不经过 CPU。

### 3.2 双缓冲设计

`FrameCapture` 维护两套缓冲（下标 0 和 1），交替使用：

```
帧 N：GL 线程向 fbo_[0] 写入（Blit），将纹理内容拷贝到 stagingBuffer_[0]
帧 N+1：编码线程读取 stagingBuffer_[0]，同时 GL 线程向 fbo_[1] 写入
```

写索引 `writeIndex_` 在 `getDevicePtr()` 被调用时翻转，从而实现捕获与编码的流水线并行，避免相互等待。

### 3.3 捕获流程（`FrameCapture::capture()`）

```cpp
// 第一步：Blit 默认帧缓冲 → 捕获 FBO
// 目标坐标 Y 轴翻转 (0, height_, width_, 0)，
// 补偿 OpenGL 纹理坐标从左下角起点的特性，使画面方向与屏幕一致
glBlitFramebuffer(
    0, 0, width_, height_,    // 来源：整个默认帧缓冲
    0, height_, width_, 0,    // 目标：Y 轴翻转后写入 FBO
    GL_COLOR_BUFFER_BIT, GL_LINEAR);

// 第二步：映射 CUDA 资源 → 2D 拷贝 → 解除映射
cudaGraphicsMapResources(1, &cudaResource_[writeIndex_], 0);
cudaGraphicsSubResourceGetMappedArray(&array, cudaResource_[writeIndex_], 0, 0);
cudaMemcpy2DFromArray(
    stagingBuffer_[writeIndex_], pitch_,   // 目标：线性暂存缓冲
    array, 0, 0,                           // 来源：已映射的 CUDA 数组
    pitch_, height_,
    cudaMemcpyDeviceToDevice);             // 全部在 GPU 内部进行
cudaGraphicsUnmapResources(1, &cudaResource_[writeIndex_], 0);
```

**为什么要 Blit 而不直接注册默认帧缓冲？**
CUDA 的 `cudaGraphicsGLRegisterImage()` 仅支持纹理对象（`GL_TEXTURE_2D`），不能直接操作 OpenGL 默认帧缓冲（它是系统创建的特殊帧缓冲，没有可供 CUDA 访问的纹理句柄）。因此必须先 Blit 到一个自己创建的 FBO。

## 四、硬件视频编码（`NvencEncoder`）

### 4.1 背景知识：NVENC 与 H.264

**NVENC**（NVIDIA Video Encode）是 NVIDIA GPU 上独立于着色器核心的专用硬件编码单元，可在不占用 CUDA 核心的前提下进行 H.264/H.265/AV1 编码。

**H.264** 是目前浏览器兼容性最广的视频编解码标准，分若干 Profile：
- **Baseline**：最低复杂度，无 B 帧，适合实时传输（WebRTC 常用）
- **Main/High**：质量更高，但延迟更大（B 帧需要未来帧，不适合低延迟）

本实现使用 **Constrained Baseline Profile** + **CBR（恒定码率）** + **超低延迟预设**，以确保 WebRTC 传输的稳定性。

### 4.2 初始化流程（`NvencEncoder::initialize()`）

```
nvEncOpenEncodeSessionEx()   // 打开编码会话，绑定 CUDA 上下文
    ↓
nvEncGetEncodePresetConfigEx(
    NV_ENC_PRESET_P3_GUID,
    NV_ENC_TUNING_INFO_ULTRA_LOW_LATENCY)  // 获取超低延迟预设
    ↓
配置关键编码参数：
  - profileGUID = NV_ENC_H264_PROFILE_BASELINE_GUID  // Baseline Profile
  - gopLength = 120                                  // 每 120 帧一个 IDR 关键帧
  - frameIntervalP = 1                               // 禁用 B 帧
  - rateControlMode = NV_ENC_PARAMS_RC_CBR           // 恒定码率
  - zeroReorderDelay = 1                             // 零重排序延迟
  - repeatSPSPPS = 1                                 // 每个 IDR 前重复序列头（SPS/PPS）
    ↓
nvEncInitializeEncoder()     // 初始化编码器
    ↓
nvEncCreateBitstreamBuffer() // 创建比特流输出缓冲区
    ↓
cudaMalloc(stagingBuffer_)   // 分配 CUDA 线性暂存缓冲
    ↓
nvEncRegisterResource()      // 将暂存缓冲注册为 NVENC 输入资源
```

**GOP（图像组）与关键帧**：H.264 视频由 P 帧（预测帧，依赖前帧）和 I/IDR 帧（关键帧，可独立解码）构成。`gopLength=120` 意味着每 2 秒（60fps）强制产生一个 IDR 帧，即使链路丢包也能在最多 2 秒内自动恢复画面。浏览器发送 PLI 信号时会立即触发一次额外的 IDR 帧，使恢复更快。

### 4.3 编码一帧（`NvencEncoder::encodeFrame()`）

```cpp
// 步骤 1：将外部线性像素数据拷贝到 NVENC 注册的暂存缓冲
cuMemcpy2D(...)  // FrameCapture 的 stagingBuffer_ → NvencEncoder 的 stagingBuffer_

// 步骤 2：映射 NVENC 输入资源（暂存缓冲 → NVENC 内部句柄）
nvEncMapInputResource(encoder_, &mapRes);

// 步骤 3：提交编码请求
nvEncEncodePicture(encoder_, &picParams);
// 若 forceIdr_ == true，编码标志中加入 NV_ENC_PIC_FLAG_FORCEIDR

// 步骤 4：解除映射，锁定比特流输出缓冲，通过回调传出 NAL 数据
nvEncLockBitstream(encoder_, &lockBs);
cb(lockBs.bitstreamBufferPtr, lockBs.bitstreamSizeInBytes, isKeyframe);
nvEncUnlockBitstream(encoder_, bitstreamBuffer_);
```

**NAL 单元**（Network Abstraction Layer Unit）是 H.264 码流的基本传输单元，每次 `encodeFrame()` 回调可能携带一个或多个 NAL，libdatachannel 的 RTP 打包器会自动将超大 NAL 切分为多个 RTP 包（FU-A 分片）。

## 五、WebRTC 信令与连接建立

### 5.1 背景知识：WebRTC 的三步握手

WebRTC 不是简单的 HTTP 流或 RTSP 推流，它需要一个"信令"过程来协商双方的编解码能力、网络地址等信息，使用 SDP（Session Description Protocol）格式：

```
浏览器                              服务器
  │                                    │
  │── 1. 创建 Offer SDP ──────────────▶│
  │      (HTTP POST /api/offer)        │
  │                                    │── 2. 解析 Offer，创建 PeerConnection
  │                                    │      选择匹配的 H.264 配置
  │                                    │      收集 ICE 候选（本地网络地址）
  │◀─────── 3. 返回 Answer SDP ─────────│
  │      (HTTP Response body)          │
  │                                    │
  │══════════════ 4. ICE 连通性检测 ═══│
  │                                    │
  │◀══════════ 5. RTP 视频流 ═══════════│  （DTLS 加密，UDP）
  │──── DataChannel 输入事件 ──────────▶│
```

SDP 包含了视频编解码方案（H.264 payload type、FMTP 参数）、媒体方向（SendOnly/RecvOnly）、SRTP 密钥材料等信息，双方通过"Offer/Answer"机制达成一致。

### 5.2 SDP 协商细节

浏览器发送的 Offer SDP 中可能包含多种 H.264 配置，例如：

```sdp
a=rtpmap:97 H264/90000
a=fmtp:97 level-asymmetry-allowed=1;packetization-mode=1;profile-level-id=42e01f

a=rtpmap:109 H264/90000
a=fmtp:109 level-asymmetry-allowed=1;packetization-mode=0;profile-level-id=42001f
```

服务器通过 `findAllH264()` + `pickBestH264()` 挑选最优配置：

| 评分项 | 分值 | 说明 |
|--------|------|------|
| `packetization-mode=1` | +100 | NVENC 输出的 NALU 无需额外处理即可用此模式分片 |
| `profile-level-id=42e0xx` | +50 | Constrained Baseline，兼容性最广 |
| `profile-level-id=42c0xx` | +40 | Baseline，略低 |
| 其他 profile | +10~30 | 降分选用 |

协商出的 Payload Type（如 97）将在整个会话中用于标识视频 RTP 包。

### 5.3 单连接强制策略

```cpp
// stream_server.cpp — /api/offer 处理函数（在 httplib 线程池中执行）
std::lock_guard<std::mutex> offerLk(offerMutex_);
// ↑ 若已有另一个窗口在建立连接，此处阻塞等待
```

当第二个浏览器窗口发送 Offer 时，`offerMutex_` 使其排在第一个处理流程之后。等第一个 `createPeerConnection()` 完成后，`offerMutex_` 释放，第二个请求进入并执行新的 `createPeerConnection()`，在其中会先 `pc_->close()` 销毁旧连接，再建立新连接。实现了"新窗口踢旧窗口"而非多窗口并发的行为。

## 六、媒体处理链（libdatachannel）

### 6.1 背景知识：RTP 与 RTCP

**RTP**（Real-time Transport Protocol）是视频/音频数据的传输协议，不保证可靠投递（基于 UDP），每个包含有序列号和时间戳用于播放同步和丢包检测。

**RTCP**（RTP Control Protocol）是 RTP 的控制协议，用于传输统计信息和控制信号：
- **SR（Sender Report）**：发送端报告发送的包数/字节数和时钟映射
- **NACK（Negative Acknowledgment）**：接收端通知发送端某个序列号的包丢失了，请求重传
- **PLI（Picture Loss Indication）**：接收端通知视频画面解码失败，请求一个完整的关键帧

### 6.2 责任链结构

`createPeerConnection()` 中构建了如下处理链：

```
NVENC 输出的裸 NAL 数据
        │
        ▼
H264RtpPacketizer        ← 将 NALU 按 MTU（~1200 字节）切分为 RTP 包，打上序列号和时间戳
        │
        ▼
RtcpSrReporter           ← 定期生成 RTCP SR 报告，发给浏览器同步时钟（影响 jitter 计算）
        │
        ▼
RtcpNackResponder        ← 缓存已发送的 RTP 包；收到浏览器 NACK 时从缓存重发该包
        │
        ▼
PliHandler               ← 收到浏览器 PLI 时，调用 encoder_.forceKeyframe()
                           下一帧将强制编码为 IDR，使浏览器立即恢复解码
```

这是标准的**责任链模式**（Chain of Responsibility）：每个环节只处理自己关注的信号类型，其余透明传递。

### 6.3 RTP 时间戳计算

```cpp
// encodeLoop 中，每个编码帧发送前计算时间戳
auto elapsed = std::chrono::duration_cast<std::chrono::microseconds>(
    now - streamStartTime_).count();
rtp->timestamp = rtp->startTimestamp +
    uint32_t(elapsed * 90000ULL / 1000000ULL);
```

H.264 的 RTP 时钟频率固定为 **90000 Hz**（90 kHz）。`elapsed` 是距会话开始的微秒数，转换公式为：

$$\text{RTP timestamp} = t_0 + \lfloor t_{\mu s} \times \frac{90000}{1000000} \rfloor = t_0 + \lfloor t_{\mu s} \times 0.09 \rfloor$$

浏览器使用此时间戳计算帧间间隔并控制播放节奏，确保视频流畅而不出现卡顿或速度飘移。

## 七、HTTP API 服务器

StreamServer 内嵌一个 httplib HTTP 服务器，监听 `0.0.0.0:8080`，提供以下端点：

| 路径 | 方法 | 功能 |
|------|------|------|
| `/` | GET | 静态文件服务（挂载 `web/` 目录，提供 `index.html`/`stream.js`） |

## 八、分辨率动态同步

### 8.1 问题背景

当用户调整浏览器窗口大小时，希望服务器的渲染分辨率与之同步，避免黑边或画质损失。但直接在 WebRTC 回调中操作 CUDA/NVENC 是不安全的（回调来自 libdatachannel 的内部线程，不持有 GL 上下文）。

### 8.2 异步 Pending Resize 机制

采用原子变量 + 延迟应用的方案，将"请求尺寸"与"实际执行"解耦：

```
浏览器端（stream.js）
  │ DataChannel 发送 {type:"resize", width:W, height:H}
  ▼
InputHandler::onMessage()     → pendingResizeW_/H_ 存储新尺寸
  ↓ 主线程消费队列
processEvents() 的 onResize 回调 → StreamServer::requestResize(w, h)
  ↓
requestResize()               → pendingW_.store(w); pendingH_.store(h);
                                （原子写，无锁，立即返回，不阻塞 GL 线程）
  ↓ 下一帧 GL 线程进入 onFrameReady()
applyPendingResize()          → std::unique_lock try_lock(pipelineMutex_)
                                若编码线程正在工作则跳过，等下一帧再试
                                否则：capture_.resize(w,h) + encoder_.cleanup() + encoder_.initialize()
                                      encoder_.forceKeyframe()
```

**为什么要 try_lock？**

`pipelineMutex_` 由编码线程在每帧编码期间持有。若 resize 在编码进行中阻塞等待，会阻塞 GL 线程，导致显示卡顿。使用 `try_lock` 失败时直接跳过本帧的 resize，下一帧再尝试，保证 GL 线程永不被锁阻塞。

## 九、输入回传（`InputHandler`）

### 9.1 事件类型与 JSON 格式

浏览器通过 DataChannel 发送 JSON 字符串，服务器解析为以下事件类型：

| 事件类型 | JSON 示例 |
|----------|-----------|
| 鼠标移动 | `{"type":"mousemove","x":0.5,"y":0.3}` |
| 鼠标按下 | `{"type":"mousedown","x":0.5,"y":0.3,"button":0}` |
| 鼠标释放 | `{"type":"mouseup","x":0.5,"y":0.3,"button":0}` |
| 滚轮 | `{"type":"wheel","dx":0,"dy":120}` |
| 键盘按下 | `{"type":"keydown","code":"KeyW","shift":false,"ctrl":false,"alt":false,"meta":false}` |
| 键盘释放 | `{"type":"keyup","code":"KeyW","shift":false,"ctrl":false,"alt":false,"meta":false}` |
| 窗口调整 | `{"type":"resize","width":1920,"height":1080}` |

鼠标坐标以**归一化坐标**（`[0,1]`）传递，在服务器端乘以窗口像素尺寸还原为像素坐标。

### 9.2 坐标系与 DPI 处理

浏览器的 CSS 像素与物理像素之间存在 `devicePixelRatio` 的缩放关系（HiDPI 屏幕通常为 2.0）。`sendResize()` 发送的是物理像素尺寸，确保服务器按实际分辨率渲染：

```javascript
// stream.js
const dpr = window.devicePixelRatio || 1;
const w = Math.max(64, (Math.round(cssW * dpr)) & ~1);  // 物理像素，偶数对齐
const h = Math.max(64, (Math.round(cssH * dpr)) & ~1);
send({ type: "resize", width: w, height: h });
```

`& ~1` 是将数值向下取偶数（NVENC 要求分辨率宽高均为偶数，这是 H.264 的 YUV 4:2:0 色度采样格式约束）。

### 9.3 键码转换

浏览器使用 `KeyboardEvent.code`（如 `"KeyW"`, `"ArrowUp"`, `"ShiftLeft"`），GLFW 使用整数键码（如 `GLFW_KEY_W = 87`）。`InputHandler::browserKeyToGlfw()` 通过一张静态哈希表完成转换，覆盖：

- 字母键 A–Z（`KeyA`–`KeyZ`）
- 数字键 0–9（`Digit0`–`Digit9`）
- 功能键 F1–F12
- 方向键、Home/End/PageUp/PageDown
- 修饰键 Shift/Ctrl/Alt/Super（左右各一个）
- 常用符号键

### 9.4 ImGui 与 GLFW 双路注入

InputHandler 同时向 ImGui 和 GLFW 的回调注入事件，确保 UI（ImGui 控件）和本地键盘处理（`keyCallback`，如 ESC 关闭、Space 暂停）均可响应远端输入：

```cpp
// 以键盘事件为例（input_handler.cpp）
io.AddKeyEvent(glfwKeyToImGuiKey(ev.key), true);    // 注入 ImGui
auto keyCb = glfwSetKeyCallback(window, nullptr);   // 临时取出 GLFW 回调
glfwSetKeyCallback(window, keyCb);                  // 立即装回
if (keyCb) keyCb(window, ev.key, 0, GLFW_PRESS, ev.mods); // 手动调用
```

鼠标事件通过 `io.AddMousePosEvent()` / `io.AddMouseButtonEvent()` 和 `io.AddMouseWheelEvent()` 注入 ImGui，ImGui 的 `_ImplGlfw_` 后端会自动同步给 GLFW 状态机。

## 十、浏览器端（`stream.js`）

### 10.1 连接建立流程

```javascript
connect()
  │
  ├ 创建 RTCPeerConnection（无 ICE 服务器，局域网直连）
  ├ addTransceiver("video", { direction: "recvonly" })  ← 声明接收视频
  ├ createDataChannel("input", { ordered: true })        ← 创建输入回传信道
  ├ createOffer() → setLocalDescription()
  ├ waitGathering(pc, 3000)    ← 等待 ICE 候选收集完成（最多 3 秒）
  ├ fetch("/api/offer", { body: localDescription.sdp, signal: AbortController.signal })
  │     ← AbortController 5 秒超时，防止服务器宕机时永久等待
  └ setRemoteDescription(answerSdp)
```

### 10.2 重连机制

```javascript
pc.oniceconnectionstatechange = () => {
    if (s === "disconnected" || s === "failed" || s === "closed") {
        showStatus("连接断开，正在重连...");
        scheduleReconnect();  // 3 秒后调用 connect()
    }
};
```

`cleanup()` 在重连前调用，除了关闭 `RTCPeerConnection` 外，还重置 `lastResizeW = 0; lastResizeH = 0`，确保重连后 DataChannel 开启时一定会重新发送 resize 事件（若不重置，去重检查会误认为"尺寸未变"而跳过，导致服务器保持旧分辨率而产生黑边）。

### 10.3 实时质量统计覆层

DataChannel 开启后，`startStats()` 每秒调用 `pc.getStats()` 读取 WebRTC 内置统计，在视频画面左上角叠加显示：

```
1920x1080  60 fps  15.2 Mbps  jitter 1.2ms  dec:ExternalDecoder
```

统计数据同时通过 `fetch("/api/browser-stats", { method: "POST" })` 上传到服务器，可通过 `/api/browser-stats` GET 端口查询（供服务端日志分析）。

## 十一、关键参数配置

### 11.1 `StreamServer::Config`

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `httpPort` | 8080 | HTTP 信令服务器端口，也用于静态文件服务 |
| `bitrateMbps` | 15 | 目标视频码率（Mbps），CBR 模式严格约束 |
| `fps` | 60 | 目标帧率，VBV 缓冲大小按此计算 |

### 11.2 `NvencEncoder::Config`

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `gopLength` | 120 | IDR 帧间隔（帧数）；`gopLength/fps` = 最长无法解码的时间窗 |
| `fps` | 60 | 用于 VBV 缓冲计算：`vbvBufferSize = bitrate / fps` |
| `bitrateMbps` | 15 | 平均码率 = 最大码率（CBR），保证带宽稳定 |

### 11.3 编码延迟来源分析

| 来源 | 典型值 | 优化措施 |
|------|--------|---------|
| NVENC 编码延迟 | ~1ms | 超低延迟预设、禁 B 帧、`zeroReorderDelay=1` |
| RTP 打包/发送 | <0.5ms | UDP 直发，无 TCP 重传延迟 |
| 网络传输（局域网） | 0.1–1ms | 无需特殊优化 |
| 浏览器解码/渲染 | 5–16ms | 硬件解码（ExternalDecoder）可显著降低 |
| **总计（局域网）** | **约 10–20ms** | — |

## 十二、编译与启用

串流功能通过预处理宏 `ENABLE_STREAMING` 条件编译，关闭时完全不影响仿真性能：

```cmake
# CMakeLists.txt
option(ENABLE_STREAMING "Build with browser streaming support" ON)
if(ENABLE_STREAMING)
    target_compile_definitions(FVM-Aerodynamic-Simulator PRIVATE ENABLE_STREAMING)
    ...
endif()
```

运行时在 `main.cpp` 中初始化：

```cpp
#ifdef ENABLE_STREAMING
StreamServer g_streamServer;
// 在 GL 上下文创建后、求解器线程启动前
g_streamServer.initialize(cfg, windowWidth, windowHeight);
// 每帧渲染完成后
g_streamServer.onFrameReady();
// 处理浏览器输入
g_streamServer.inputHandler().processEvents(window, windowWidth, windowHeight, onResize);
#endif
```

启动程序后，在局域网内任意设备的浏览器中访问 `http://<服务器IP>:8080` 即可开始串流。
